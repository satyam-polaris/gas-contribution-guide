Root cause

Why we're seeing 40–50 commits when we merge to staging

When we use GitHub's "Sync Fork" button, it creates a merge commit that pulls all upstream history into our fork. Those commits get brand-new SHA identifiers in our fork even though the code is identical. When we then open a PR from upstream development → staging, Git treats all of them as new commits and dumps them into the PR diff.

What goes wrong — the SHA explosion

Three things combine to cause this:

  1. We're using "Sync Fork" instead of rebase — that button does a merge under the hood, creating extra commits.
  2. Staging has drifted from development — the longer we leave the gap, the bigger the commit flood when we finally merge.
  3. We're not squashing PR commits — every WIP commit piles up and makes the history unreadable.
The fix Rebase instead of merge when syncing our forks. Squash commits in PRs. Promote development → staging → prod regularly, not in big batches.

Syncing our forks

The right way to sync our forks with upstream

❌ Never do this Using GitHub's "Sync Fork" button does a merge under the hood and creates extra commits that flood our PR history. We never use that button.
✅ We always rebase from upstream instead

One-time setup — add the upstream remote

Every team member needs to run this once after forking:

# Add upstream as a remote (run once)
git remote add upstream https://github.com/UPSTREAM-ORG/REPO.git

# Verify we have both remotes
git remote -v
# origin    https://github.com/YOUR-NAME/REPO.git  (our fork)
# upstream  https://github.com/UPSTREAM-ORG/REPO.git  (the main repo)

Syncing our fork's development branch (do this regularly)

# 1. Fetch upstream changes — this doesn't touch our code yet
git fetch upstream

# 2. Switch to our fork's development branch
git checkout development

# 3. Rebase onto upstream/development (NOT merge)
git rebase upstream/development

# 4. Push to our fork — force-with-lease is safe after rebase
git push origin development --force-with-lease
⚠️ First time syncing after using "Sync Fork" button If we've already used the button and made a mess, do a one-time reset — this is safe because our feature work lives in separate branches, not on development directly. 
git fetch upstream
git checkout development
git reset --hard upstream/development
git push origin development --force-with-lease

PR flow

Our full pull request flow — fork to upstream/development

PR lifecycle — feature branch to upstream/development

Step-by-step commands

Sync our fork before starting any work

We always do this first — it keeps our fork up to date before we branch off.

git fetch upstream
git checkout development
git rebase upstream/development
git push origin development --force-with-lease

Create our feature branch

Always branch from development — never from staging or prod.

git checkout development
git checkout -b feature/your-feature-name

Do our work and commit freely

WIP commits are totally fine — they'll be squashed into one commit when the PR merges.

git add .
git commit -m "WIP: scaffold auth structure"
git commit -m "WIP: add JWT validation logic"

Push our branch and open a PR

We push to our fork, then open the PR targeting upstream/development — not our fork's development.

git push origin feature/your-feature-name

# On GitHub: open PR from
# YOUR-FORK/feature/your-feature-name → UPSTREAM/development

Use Squash & Merge — always

In GitHub's merge dropdown, we always select "Squash and merge". Write a proper commit message:
feat: add JWT auth module (#42)

Delete the feature branch after merging

# GitHub shows a "Delete branch" button — click it
# Or from terminal:
git push origin --delete feature/your-feature-name
git branch -d feature/your-feature-name

Promoting to staging & prod

How we promote: development → staging → prod

This is handled via PRs on the upstream repo — usually by the team lead or whoever owns the deployment. We never push directly to upstream/staging or upstream/prod.

Promotion flow — three environments

Promoting development → staging

This is a PR on the upstream repo. The team lead or release owner does this every 1–2 days — or whenever a meaningful batch of features has landed on development.

# Team lead: raise a PR on the upstream repo
# Base: upstream/staging ← Compare: upstream/development
# Title: "Promote development to staging - YYYY-MM-DD"
# Use squash merge here too

# After the PR merges, tag it:
git fetch upstream
git checkout staging
git pull upstream staging
git tag staging-2026-04-02
git push upstream staging-2026-04-02

Promoting staging → prod (after QA sign-off)

# Team lead: raise a PR on the upstream repo
# Base: upstream/prod ← Compare: upstream/staging
# Title: "Release v1.4.0"
# Requires QA approval in the PR before merging

# After the PR merges, tag the release:
git fetch upstream
git tag v1.4.0 upstream/prod
git push upstream v1.4.0
⚠️ Never let staging lag development by more than 3 days The longer we wait, the more commits pile up and the harder it is to review what's changing. Small frequent promotions are much safer than big batches.

Prod branch sync

Keeping our fork's prod branch in sync — and debugging on prod

Sometimes we need to check out the prod branch on our fork to investigate a live issue. Our fork's prod branch is read-only and never pushed from — it exists purely so we can inspect what's running in production locally.

🔴 The golden rule for fork/prod

We never commit to or push from our fork's prod branch. We only pull it in sync with upstream/prod. Any fixes we discover while inspecting prod go into a feature branch off development — never direct commits to prod.

Setting up tracking for upstream/prod (one-time)

# Make sure we have the upstream remote
git fetch upstream

# Create our local prod branch tracking upstream/prod
git checkout --track upstream/prod -b prod

# Verify it's tracking upstream
git branch -vv
# Should show: prod ... [upstream/prod]

Syncing fork/prod before any debugging session

Every time we want to look at prod, we sync first — otherwise we're looking at stale code.

# Always do this before checking prod
git fetch upstream

# Hard reset our local prod to match upstream/prod exactly
git checkout prod
git reset --hard upstream/prod

# Confirm we're in sync
git log --oneline -5
# Should match what's on the upstream/prod branch

Checking the diff between prod and staging

# What's on staging that hasn't hit prod yet?
git log upstream/prod..upstream/staging --oneline

# What's the actual code diff?
git diff upstream/prod..upstream/staging

# What's on dev that hasn't hit staging yet?
git log upstream/staging..upstream/development --oneline

If we find a bug while on prod — what to do

Prod bug investigation flow
# We found a bug. Here's what we do:

# 1. Note exactly what file/line we found the issue on prod
git log --oneline -3    # note the commit hash for reference

# 2. Switch back to development and sync it
git checkout development
git fetch upstream
git rebase upstream/development

# 3. Create a fix branch from development
git checkout -b fix/null-pointer-user-service

# 4. Apply the fix, commit, and open a normal PR to upstream/development
git commit -m "fix: resolve null pointer in user service"
git push origin fix/null-pointer-user-service

# 5. After it merges to development, the team lead fast-tracks it
# through staging → prod (same PR promotion process, just faster)
💡 For critical hotfixes that can't wait for the normal dev → staging → prod flow Talk to the team lead. They can raise an emergency PR directly from a hotfix branch to staging → prod, bypassing development. This is the exception, not the rule — and still requires a review.

Team rules

Rules we all follow

✅ We do
🔀 Squash & Merge all PRs into upstream/development
🔄 Sync our forks via rebase, never the "Sync Fork" button
📅 Promote development → staging every 1–2 days
🌿 Branch from development for all features and fixes
🏷️ Tag releases when we promote to prod (v1.x.0)
✍️ Write clear squash commit messages (feat/fix/chore: ...)
🔴 Sync fork/prod before any debugging session on prod
🗑️ Delete our feature branch after it merges
❌ We never do
🚫 Use GitHub's "Sync Fork" button — ever
🚫 Commit or push directly to upstream/staging or upstream/prod
🚫 Push anything from our fork/prod branch
🚫 Use "Create a merge commit" in the PR merge dropdown
🚫 Let staging lag development by more than 3 days
🚫 Keep feature branches open for more than 1 week
🚫 Branch features from staging or prod
🚫 Force push to upstream/development, staging, or prod

Commit message format

We use conventional commits for our squash messages — it makes the changelog automatic and readable:

feat: add JWT authentication module (#42)
fix: correct null pointer in user service (#51)
chore: update dependencies to latest (#55)
docs: update README setup instructions (#48)
refactor: extract payment logic into service (#60)
BREAKING CHANGE: rename /api/v1 to /api/v2 (#70)

Checklist

Our PR checklist — tick before we submit

  • We synced our fork with upstream/development before creating this branch (git fetch upstream + git rebase upstream/development)
  • Our feature branch was created from development — not from staging or prod
  • Our PR targets upstream/development — not our fork's development
  • We have no merge conflicts with upstream/development
  • We will use Squash and merge — never "Create a merge commit"
  • Our squash commit message uses conventional format: feat/fix/chore: description (#PR)
  • We have at least 1 reviewer approval before merging
  • We will delete our feature branch after the PR merges
  • We have NOT used the "Sync Fork" button anywhere in this workflow

Troubleshooting

FAQ & troubleshooting

🔥 Our staging PR shows 40–50 commits. What do we do?

We need a one-time cleanup. First make sure all open feature branches are merged, then:

# On the upstream repo — reset staging to match development exactly
git fetch upstream
git checkout staging
git reset --hard upstream/development
git push upstream staging --force-with-lease

# Staging and development are now in sync
# Future PRs will only show new commits going forward
⚠️ Coordinate with the team first Before force-pushing to upstream/staging, post in the team chat: "Resetting staging to match development — no pushes until I confirm done."

🤔 What if someone already used "Sync Fork" on their fork?

# Reset our fork's development to match upstream/development
git fetch upstream
git checkout development
git reset --hard upstream/development
git push origin development --force-with-lease

# Our feature branches are untouched — they're safe
# Rebase any open feature branches onto the clean development
git checkout feature/our-branch
git rebase origin/development

🔴 Our fork/prod is out of date — how do we re-sync it?

# Fetch the latest from upstream
git fetch upstream

# Hard reset our local prod to exactly match upstream/prod
git checkout prod
git reset --hard upstream/prod

# Optional: update our fork's remote prod too
git push origin prod --force-with-lease

# Confirm sync
git diff upstream/prod HEAD  # should show nothing
