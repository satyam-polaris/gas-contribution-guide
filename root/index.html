Root cause

Why you're seeing 40–50 commits on staging

When you click GitHub's "Sync Fork" button, it creates a merge commit that pulls in all upstream history. Those commits get new SHA identifiers in your fork. When you eventually open a PR to staging, Git sees them as brand-new commits — even though the code is identical — and dumps all of them into the PR diff.

Root cause — commit SHA explosion

Three things make this worse together:

  1. Sync Fork creates merge commits — GitHub's button does a merge, not a rebase.
  2. staging drifted from development — letting them diverge for days/weeks means a huge catch-up dump later.
  3. No commit squashing — every tiny WIP commit piles up into the PR history.
The fix is simple Rebase instead of merge when syncing. Squash commits in PRs. Merge development → staging frequently (not in big batches).

3
Fork syncing

The right way to sync your fork

❌ Never do this Using GitHub's "Sync Fork" button does a merge under the hood and creates extra commits that pollute your PR history.
✅ Do this instead — rebase from upstream

One-time setup (add upstream remote)

# Run this once after forking
git remote add upstream https://github.com/ORIGINAL-OWNER/REPO.git

# Verify you have both remotes
git remote -v
# origin    https://github.com/YOUR-NAME/REPO.git (fetch)
# origin    https://github.com/YOUR-NAME/REPO.git (push)
# upstream  https://github.com/ORIGINAL-OWNER/REPO.git (fetch)
# upstream  https://github.com/ORIGINAL-OWNER/REPO.git (push)

Syncing (do this regularly — not in big batches)

# 1. Fetch upstream changes (doesn't touch your code yet)
git fetch upstream

# 2. Switch to your main branch
git checkout main

# 3. REBASE (not merge!) onto upstream/main
git rebase upstream/main

# 4. Push to your fork (force-push is OK here since we rebased)
git push origin main --force-with-lease
⚠️ First time syncing after using "Sync Fork" button Your history is already messy. You'll need to do a one-time reset: git reset --hard upstream/main then force push. This is safe because your feature work is in separate branches — main should always mirror upstream.

After rebasing main — propagate to staging and development

# Bring staging up to date
git checkout staging
git rebase main   # or: git merge main
git push origin staging

# Bring development up to date
git checkout development
git rebase staging
git push origin development

PR flow

The complete pull request flow

PR lifecycle — from feature to production

Step-by-step commands

Create your feature branch

Always branch from development, never from staging or main.

git checkout development
git pull origin development
git checkout -b feature/your-feature-name

Do your work and commit freely

WIP commits are fine here — they'll be squashed later.

git add .
git commit -m "WIP: setting up auth structure"
# ... keep working ...
git commit -m "WIP: add JWT validation"

Push and open a PR

PR goes from feature/xyz → development (on the upstream repo).

git push origin feature/your-feature-name
# Then open PR on GitHub: feature branch → upstream/development

Squash & Merge (IMPORTANT)

When merging the PR, always select Squash and merge in GitHub's dropdown. Write a good commit message like: feat: add JWT auth module (#42)

Sync development → staging (regularly!)

Don't let them drift. Do this every 1–2 days or after every significant feature lands.

git checkout staging
git merge development   # or use a PR
git push origin staging

Team rules

Rules every team member must follow

✅ Do
🔀 Squash & Merge all PRs into development
🔄 Sync fork via rebase (never "Sync Fork" button)
📅 Merge development → staging every 1–2 days
🌿 Branch from development for all features
🏷️ Tag releases with v1.x when promoting to main
✍️ Write meaningful squash commit messages
❌ Don't
🚫 Never use GitHub "Sync Fork" button
🚫 Never commit directly to staging or main
🚫 Never let staging lag development by > 3 days
🚫 Never use "Create a merge commit" in PR dropdown
🚫 Never keep feature branches open for > 1 week
🚫 Never force push to shared branches (staging/main)

Commit message format

Use conventional commits for squash messages — this makes the changelog readable:

feat: add JWT authentication module (#42)
fix: correct null pointer in user service (#51)
chore: update dependencies to latest (#55)
docs: update README setup instructions (#48)
refactor: extract payment logic into service (#60)
BREAKING CHANGE: rename /api/v1 to /api/v2

GitHub settings to enforce these rules

Ask your repo admin to configure these settings on the upstream repo:

Setting Where Value
Allow squash merging Settings → General → Pull Requests ✅ Enabled
Allow merge commits Settings → General → Pull Requests ❌ Disabled
Allow rebase merging Settings → General → Pull Requests ❌ Disabled (optional)
Require PR reviews Settings → Branches → Branch rules ✅ 1 approval minimum
Require branches to be up to date Settings → Branches → Branch rules ✅ Enabled
Delete branches after merge Settings → General → Pull Requests ✅ Enabled

Troubleshooting

FAQ & troubleshooting

🔥 My staging PR shows 50 commits. What do I do?

You need a one-time cleanup. First, make sure all feature branches are merged. Then:

# Reset staging to exactly match development
git checkout staging
git fetch origin
git reset --hard origin/development
git push origin staging --force-with-lease

# Now staging and development are in sync — future PRs will be clean
⚠️ Coordinate with your team before force-pushing to staging Tell everyone in Slack: "I'm resetting staging to match development — don't push until I confirm done."

🤔 What if I already used "Sync Fork" and made a mess?

# On your fork's main branch — reset it to upstream
git fetch upstream
git checkout main
git reset --hard upstream/main
git push origin main --force-with-lease

# Your feature branches are safe — they're untouched
# Rebase any open feature branches onto the cleaned main
git checkout feature/your-branch
git rebase origin/main
